# ADR-001: Verdict + risk as machine-readable files; a distinct review stage

| Field    | Value             |
| -------- | ----------------- |
| Status   | accepted          |
| Date     | 2026-06-24        |
| Deciders | platform-dx       |
| Related  | SPEC-001, PRD-001 |

## Context

The readiness review must (a) be able to halt the loop deterministically and
(b) keep the maker/checker separation the constitution requires. A gate enforced
by asking the model nicely in a prompt is exactly what CON-030 forbids.

## Decision

We will:

1. Add a **distinct `spec_review` stage** owned by a new read-only
   `spec-reviewer` agent — separate from `architect` (which authored the spec),
   preserving maker/checker separation.
2. Have the stage write **machine-readable outputs** the controller reads:
   `spec-review.verdict` (`READY|CAVEATS|NOT_READY`) and
   `spec-review.riskclass` (`low|standard|sensitive`), alongside the
   human-readable `spec-review.md` scorecard. The controller — not the model —
   decides whether to halt.
3. Use **hard boundaries**: an explicit list of critical-gap categories that
   force `NOT_READY`, so the reviewer cannot rubber-stamp.
4. Make the stage **fail-open when disabled** (`spec_review.enabled: false`) and
   default the verdict to `READY` if no verdict file is produced, so the feature
   is backward compatible and never wedges a run on a missing artifact.

## Consequences

**Positive**

- Deterministic gate; maker/checker preserved; backward compatible.
- The "start here" + Critical/Optimization split makes output prioritized, not a
  flat wall of nits.

**Negative / trade-offs**

- One more stage and model call per run (cheap relative to implement/review).
- Fail-open means a silently broken reviewer degrades to today's behavior rather
  than failing loudly; mitigated by surfacing "verdict defaulted" in the report.

**Neutral**

- Risk classification starts heuristic; it can later be learned.

## Alternatives considered

- **Fold the check into `spec-init`** — rejected: the author and the checker
  would be the same agent, breaking maker/checker separation.
- **Enforce the verdict via the model's prompt** — rejected: violates CON-030
  (gates are scripts/hooks, not model instructions).
