# ADR-001: One setup script, two modes; consent before any global change

| Field    | Value             |
| -------- | ----------------- |
| Status   | accepted          |
| Date     | 2026-06-24        |
| Deciders | platform-dx       |
| Related  | SPEC-004, PRD-004 |

## Context

We need a setup that is friendly when interactive and safe when scripted, without
maintaining two divergent tools, and without surprising anyone by mutating their
system.

## Decision

We will:

1. Ship a single `scripts/setup.sh` with modes: interactive (default), `--yes`
   (non-interactive, safe defaults), `--check` (status only), `--brief`. Make
   `install.sh` a thin delegate to `setup.sh --yes` so CI keeps working.
2. **Never auto-install system packages non-interactively.** In `--yes`, system
   installs are printed, not run; only safe local actions (chmod, smoke test) run.
   Interactively, every install command is shown before it is offered to run.
3. **Behave non-interactively when stdin is not a TTY** (read defaults instead of
   blocking), so the script is safe in pipes and CI.
4. Reuse the existing primitives: detect stacks via `adapters/detect.sh`, smoke-
   test via `make selftest` (no model, no cost), and end with tailored next steps.

## Consequences

**Positive**
- One tool, two audiences; safe by default; fast for pros.

**Negative / trade-offs**
- Package-manager / tool install hints are best-effort per OS and may need updates
  as ecosystems change.

**Neutral**
- `doctor.sh` remains the read-only status reporter; setup is the guided installer.

## Alternatives considered

- **Auto-install everything silently** — rejected: mutating a developer's machine
  without consent is a non-starter (PRD: no surprises).
- **Two scripts (interactive + CI)** — rejected: divergence and duplication; one
  script with modes is simpler.
