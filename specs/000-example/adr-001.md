# ADR-001: Integer seconds, explicit errors, zero dependencies

| Field    | Value             |
| -------- | ----------------- |
| Status   | accepted          |
| Date     | 2026-06-24        |
| Deciders | platform-dx       |
| Related  | SPEC-000, PRD-000 |

## Context

The duration parser will be vendored into many tools across several languages.
It must behave identically everywhere and must not drag in dependencies. Inputs
come from humans and configs, so malformed input is expected and must fail
loudly rather than silently.

## Decision

We will:

1. Return a **whole number of seconds** as an integer. Reject fractional input
   rather than rounding (rounding rules would differ across tools).
2. Surface **explicit, typed errors** — `invalid input`, `unknown unit`,
   `overflow` — instead of returning a sentinel like `0` or `-1`.
3. Implement the core as a **pure function with no dependencies**, single-pass
   over the input. Any CLI wrapper is a thin, separate layer.
4. Support exactly the units `w,d,h,m,s`. Calendar units (`M`, `y`) are excluded
   because their second-length is ambiguous.

## Consequences

**Positive**

- Deterministic, portable behavior; trivial to port and test.
- Overflow and typos fail fast and visibly.

**Negative / trade-offs**

- No fractional durations (`1.5h` must be written `1h30m`).
- Callers needing calendar math must handle it themselves.

**Neutral**

- The unit set is fixed; adding a unit is a new ADR.

## Alternatives considered

- **Return a float of seconds** — rejected: cross-language float formatting and
  rounding differences reintroduce the inconsistency we are removing.
- **Clamp on overflow instead of erroring** — rejected: silently wrong values
  are worse than a loud failure (PRD non-goal: no silent truncation).
- **Accept calendar units (`M`,`y`)** — rejected: ambiguous second-length;
  out of scope per SPEC §2.
