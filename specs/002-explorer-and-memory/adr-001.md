# ADR-001: Local markdown memory; explorer as a cheap pre-plan stage

| Field    | Value             |
| -------- | ----------------- |
| Status   | accepted          |
| Date     | 2026-06-24        |
| Deciders | platform-dx       |
| Related  | SPEC-002, PRD-002 |

## Context

We need cross-run memory and a codebase recon step without adding services,
dependencies, or unbounded cost, and without breaking the existing 8-stage flow
or the local/auditable data-handling rules.

## Decision

We will:

1. Store memory and backlog as **local markdown** under `.loop/` (gitignored),
   not in an external store. The loop reads the path at start and appends a small
   **digest** (not full transcripts) at end, keeping growth bounded and secrets
   local (CON-090..092).
2. Run the explorer as a **distinct, cheap stage** (`explore`) before `plan`,
   using the `explore` model tier (haiku) — reconnaissance, not reasoning. It is
   **read-only** (maker/checker hygiene; it never edits).
3. Make `make clean` preserve memory/backlog (remove only per-run artifacts) and
   add `make clean-all` for a full reset — so cross-run memory actually survives
   the routine cleanup.
4. Have stages declare deferred items via a per-run file the controller carries
   into the persistent backlog — the controller owns persistence, not the model.

## Consequences

**Positive**

- No new dependencies/services; fully local and auditable.
- Plans are grounded; deferred work persists; cost stays low.

**Negative / trade-offs**

- Markdown memory is simple, not queryable like a DB.
- Another stage and a cheap model call per run.

**Neutral**

- Memory schema is informal; can be formalized later if needed.

## Alternatives considered

- **External memory (Linear/DB via MCP)** — deferred to Tier 2 connectors;
  heavier and not local-first.
- **Fold explore into plan** — rejected: muddies the cheap-recon vs.
  expensive-reasoning split and the per-stage model tiers.
